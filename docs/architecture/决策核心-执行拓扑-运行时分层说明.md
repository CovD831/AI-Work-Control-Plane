# 决策核心、执行拓扑、运行时分层说明

## 目的

这份文档用于明确本项目后续实现时的概念边界，避免以下几类东西继续混在一起：

- 决策核心
- `agent team`
- `claude / codex` 之类的委派执行
- provider / runtime / job backend

从现在开始，项目统一按 **AI Work Control Plane + 三层结构** 理解和推进。

## 顶层：AI Work Control Plane

新的产品重心不是“让 agent 更像人类公司”，也不是继续堆叠显式编排花样，而是为 AI 工作构造更适合模型效率的外部控制面：

```text
WorkspaceState -> ContextPacket -> StrategyDecision -> ExecutionTopologySnapshot -> ApprovalItem -> EvidenceBundle -> MemoryRecord
```

这一层负责：

- 维护真实 workspace state，而不是只维护一次会话状态
- 把文档、变更、记忆、风险压缩成最小充分上下文
- 记录 strategy decision，但不让 context packet 自行决策
- 把 approved plan / work graph / approval / evidence 编译成只读 topology snapshot
- 把人类介入做成 `ApprovalItem`，解析和记录决策，但不绕过 execution gate
- 把测试、合规、evidence report、setup doctor 汇成 `EvidenceBundle`
- 给 memory 记录 provenance、freshness、confidence 和 optional external cache 状态

因此，下面三层仍然保留，但它们从“产品天花板”降级为 Work Control Plane 的实现层。

时间尺度上也要保持清楚：短期靠显式 agent 编排解决真实工作；中期由 AI Work Control Plane 管住编排、证据和恢复；长期即使模型逐步内化规划、委派、审核和工具使用，workspace state、evidence、approval、memory provenance 和 recovery 仍然留在系统外部，不能消失在模型私有推理里。

## 三层结构

### 1. 决策核心层

这一层是项目最稳定的核心。

它负责：

- 任务接收与任务约束解释
- planning governance
- 用户与计划主控的 `intake_chat -> draft_ready -> awaiting_human_confirmation` 阶段门禁
- 上下文管理与上下文同步约束
- 对抗性审核策略
- 执行门禁
- agent 角色、artifact、job sandbox 的硬权限约束
- role contracts、allowed actions、required outputs
- route / reroute / rescue / stop 决策
- 计划产物、执行产物、审核产物之间的关联
- approval state、human intervention reason、usage/cost placeholder

一句话：

> 决策核心层负责决定“该不该做、怎么做、做到什么程度才能继续往下走”。

当前仓库里更接近这一层的内容包括：

- `PlanSession`
- `RoundController`
- `guards.py`
- gap closure
- approved-plan gate
- execution strategy decision contract

### 2. 执行拓扑层

这一层是 **可热插拔的**。

它负责：

- 任务由谁协作完成
- 多个 agent 之间如何分工
- 用户确认首版计划后，Lead 与 adversarial reviewer 如何围绕同一计划质疑和补全
- 是否带 reviewer / adversarial reviewer
- 是否走 team / cluster / swarm / solo
- revision loop 以什么协作结构推进

一句话：

> 执行拓扑层负责决定“任务用什么 agent 协作形态来完成”。

这里的典型形态包括：

- `solo`
- `team`
- `team_with_adversarial_review`
- `cluster`

因此：

- `agent team` 属于执行拓扑层
- 它不是整个产品本身
- 它是一种可以被替换、组合、比较的协作拓扑

当前仓库里更接近这一层的内容包括：

- `TeamOrchestrator`
- team chat / review / adversarial review / revision round 工作流
- task pool / next executable task
- lightweight main/review/rescue thread visibility
- decisions / lessons / workflow notes knowledge artifacts

### 3. Provider / Runtime 层

这一层同样是 **可热插拔的**。

它负责：

- 底层具体调用哪个模型或工具执行
- 用什么 command runtime / background session / job backend 落地
- Claude、Codex、mock 等 provider 如何接入
- 在 job 启动边界二次执行 sandbox guard，防止 review/research 类任务获得写权限

一句话：

> Provider / Runtime 层负责决定“具体由谁执行，以及怎样把执行过程跑起来并记录下来”。

这里的典型模块包括：

- `claude`
- `codex`
- `mock`
- `CommandJobRuntime`
- jobs store
- command / bridge / local runtime

因此：

- “委派 cc / Claude” 更适合归到这一层
- 它不是 `agent team` 的同义词
- 它更像是执行后端或 provider-backed runtime

当前仓库里更接近这一层的内容包括：

- `ClaudeCodeAdapter`
- `CodexCliAdapter`
- `CommandJobRuntime`
- `FileJobRuntime`
- `fresh` / `resume` / `resume_if_same_task` execution context policy
- per-job runtime health and usage/cost placeholder metadata

### 补充：UI 作战台不单独算第四层

本轮新增的 terminal cockpit / dashboard 前端，不应被理解成新的“产品核心层”。

它更适合作为：

- 决策核心层的可视化观察面
- 执行拓扑层的协作态展示面
- Provider / Runtime 层的终端与 job 状态操作面

也就是说：

- UI 负责把三层已经存在的语义和状态暴露出来
- UI 不负责重新定义这三层的规则
- 如果一个能力只有“页面上好看了”，但没有对应到三层中的真实语义边界，那么它不应成为新的系统抽象

当前仓库里对应这部分的实现主要包括：

- `ui_server.py`
- `ui_service.py`
- `ui_frontend/`
- `ui_static/` 作为回退壳

## 三层之间的关系

项目不应再被理解成：

`项目 = agent team`

也不应被理解成：

`项目 = 某个 provider 的壳`

更准确的关系应该是：

`项目 = 决策核心层 + 可热插拔执行拓扑层 + 可热插拔 Provider / Runtime 层`

也就是说：

- 决策核心层决定规则
- 执行拓扑层决定协作方式
- Provider / Runtime 层决定具体调用谁、如何运行

## 当前执行层的准确定义

当前仓库中的执行层，更准确地说是：

> 一个以工作流治理为核心、带有多角色语义和有限 agent 分工的 orchestration runtime。

这意味着：

- 它不是简单脚本工作流，因为它已经有 handoff、review、adversarial review、work graph、resume、approval、evidence 和 recovery
- 它也不是高自治 multi-agent system，因为主要控制权仍然集中在主编排器与显式阶段门禁上
- `lead / reviewer / adversarial_reviewer / builder` 当前更像流程中的角色槽位，而不是拥有独立目标、预算、协商权和局部策略循环的自治 agent

后续如果要向更强多 agent 协作演进，建议保持以下顺序：

1. 先结构化协作对象
2. 再外显决策候选与裁决理由
3. 再增强局部自治和非线性交互

不能反过来先堆复杂协作花样，否则会削弱审计与恢复能力。

## 组合方式

后续目标不是只支持一种固定结构，而是允许组合，例如：

- `decision core + solo + codex`
- `decision core + team + claude`
- `decision core + team + codex implementation + claude review`
- `decision core + cluster + mixed providers`

项目真正要保持稳定的，是决策核心层的规则与治理语义，而不是某一个固定的 agent 形态或 provider。

## 后续实现的归类规则

从现在开始，新增能力进入仓库前，先判断它属于哪一层：

### 归到决策核心层

如果它主要决定以下事情：

- 是否允许执行
- 如何规划
- 如何审核
- 如何管理上下文
- 如何判断计划和执行产物是否合格

那么它属于 **决策核心层**。

### 归到执行拓扑层

如果它主要决定以下事情：

- agent 之间怎么协作
- 是否需要 team / reviewer / adversarial reviewer / cluster
- revision loop 由谁推进

那么它属于 **执行拓扑层**。

### 归到 Provider / Runtime 层

如果它主要决定以下事情：

- 调用 Claude 还是 Codex
- 用什么命令行接口
- 用什么 job runtime / backend / bridge
- 如何持久化 job 状态与命令执行结果

那么它属于 **Provider / Runtime 层**。

## 对当前仓库的直接结论

当前仓库里可以先按下面这套方式理解：

- `PlanSession`、`RoundController`、gap closure、approved-plan handoff：
  属于 **决策核心层**

- `TeamOrchestrator`：
  属于 **执行拓扑层**

- `ClaudeCodeAdapter`、`CodexCliAdapter`、`CommandJobRuntime`、`FileJobRuntime`：
  属于 **Provider / Runtime 层**

## v1 聊天式协作流

当前 v1 不再把作战指挥室作为默认产品形态。标准流是：

1. `intake_chat`：用户和 Lead 多轮聊天，Lead 起草并更新草案。
2. `draft_ready`：用户确认第一版计划可以被审查。
3. `adversarial_review`：Lead 发起 review / adversarial review，reviewer 只能产出 findings。
4. `awaiting_human_confirmation`：系统回到用户，等待补充、修订或批准。
5. `approved_for_execution`：只有人类批准后才生成 approved plan / execution contract。
6. `executing`：builder 从 approved plan 执行，不允许从 raw requirement 绕过门禁。

底层可以继续使用 JSONL message 和结构化 artifact，但 UI 首先呈现人类可读的聊天流。

## 使用建议

后续推进项目时，应尽量遵守以下原则：

1. 不要再把 `agent team` 直接当成产品核心本体。
2. 不要把 Claude / Codex 这类 provider 和 team / cluster 这类协作拓扑混成一个概念。
3. 先稳定决策核心层，再扩展执行拓扑层，最后再增加更多 provider / runtime 组合。
4. 当一个能力看起来“既像流程、又像执行、又像模型调用”时，优先拆开，而不是继续堆在一个模块里。

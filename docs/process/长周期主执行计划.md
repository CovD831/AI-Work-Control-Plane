# 长周期主执行计划

## 目的

这份文档用于把本项目的推进方式，从“每个小阶段单独起一个计划”切换为一份可长期连续执行的工程程序。

目标不是一次性做成 near-v1，而是先把当前仓库推进到：

> `internal default`

也就是：

- 默认通过 `decision core + fixed dual-model review + agent team execution`
- 来持续推进当前仓库
- 尽量减少每一个小动作都要重新确认

## 默认工作原则

从这份文档生效开始，后续默认按以下顺序推进：

1. 先补 `AI Work Control Plane` 的 artifact pipeline
2. 再补 `决策核心层`
3. 再补 `执行拓扑层`
4. 最后补 `Provider / Runtime 层`

对应三层分工说明见：

- [决策核心-执行拓扑-运行时分层说明](/Users/abab/Desktop/AI-Work-Control-Plane/docs/architecture/决策核心-执行拓扑-运行时分层说明.md)

## 当前主目标

当前主目标不是追求产品完整度，而是让这套系统先成为当前仓库内部的默认推进方式。新的重心是 **AI Work Control Plane**：原有编排不舍弃，但从产品核心下沉为 control plane 的执行能力。

也就是说，优先级是：

1. 当前仓库可持续使用
2. 当前仓库内部默认可用
3. 再考虑更完整的产品化与对外化

当前执行基准：

- 以这份长周期主计划作为唯一主线，不再按“小计划一轮轮确认”运行
- 每个实现段都必须先做定点自检，再做全量验证
- 验证通过后自动进入下一段
- 普通进展汇报不构成停点，也不等待额外一句“继续做”

## 阶段程序

### 阶段 1：补齐可恢复的 planning governance

目标：

- 把决策核心从“能拦截”推进到“能裁决、能修订、能恢复、能继续”

主要内容：

- 固定双模型 author/reviewer 对抗审查
- `DecisionVerdict`
- `team revise`
- gap closure
- required gap / optional follow-up 区分
- `resume` 的恢复语义
- `status` 的下一步动作提示

退出条件：

- 一个 session 可以完整走完：
  - `start -> needs_revision -> revise -> approve -> execute`
- 中断后能恢复
- 不再需要手工改 session 字段推进流程

### 阶段 2：把 Claude delegation 变成默认可用协作者

目标：

- 降低人工盯防 review / adversarial review 的频率

主要内容：

- team review / adversarial review 默认可接 `claude`
- delegated job 的状态可见
- delegated job 失败时给出明确下一步建议
- 保留 `mock` 与 `command + claude` 可切换能力

退出条件：

- 可以稳定使用 `command + claude` 跑 team review
- 不需要翻底层 job 文件才能判断下一步

### 阶段 3：让 approved plan 真正驱动 execution

目标：

- 摆脱 execution 仍然从 raw requirement 起跑的旧路径

主要内容：

- approved plan 成为 execution 正式输入
- execution provenance 记录 selected topology / provider/runtime / decision rationale
- run artifact 记录 approved-plan provenance
- team execution 与 direct execution 明确区分

退出条件：

- team execution 都能追溯到 approved plan
- run artifact 中清楚记录来源 session 和 approved plan

### 阶段 4：形成默认内部开发工作流

目标：

- 把当前仓库切换到“默认用这套系统推进自己”

推荐 happy path：

- `team start`
- `team status`
- `team revise`
- `team approve`
- `team execute`

退出条件：

- 不需要频繁打开 JSON 文件理解当前状态
- 大多数当前仓库任务都能按 happy path 连续推进
- 人工确认降到里程碑级别，而不是每个小动作都确认

当前阶段默认连续施工队列：

- 把 delegated review / adversarial review 的恢复入口做成标准 `team <action>` 命令族
- 把 `team next` 从单条命令建议推进成更完整的 operator runbook
- 把文档同步 / compliance / hook blocking 接入当前 happy path，而不是等到单独文档阶段才开始

## 允许中断主程序的情况

只有以下情况，才应该停止当前主程序并重新规划：

- 要修改产品定位
- 要重定三层边界
- 要引入新的执行拓扑主形态
- 要替换默认 provider/runtime 路线
- 某阶段退出条件连续无法满足

除此之外，默认继续沿当前阶段推进。

## 当前默认解释

以后当用户只说：

- `继续推进`
- `按主计划继续`
- `继续做`

默认意味着：

- 继续当前未完成阶段
- 若当前阶段退出条件满足，则进入下一阶段
- 不再要求重新拆一轮新的大方向计划
- 若当前实现段验证通过，则自动进入当前阶段的下一实现段

## 当前定位

这份文档是当前仓库的 **中文内部执行章程**。

它的用途不是替代 roadmap，而是：

- 给连续实现提供稳定的程序化依据
- 降低每轮重新规划的成本
- 让后续实现更接近“长期执行”而不是“单步演示”

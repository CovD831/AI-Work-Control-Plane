# 完整计划待办清单

## 目的

这份文档用于说明当前仓库从 `internal default` 推进到完整 roadmap v1 后的状态、验收清单和后续 v1.x backlog。

它不是新的主计划，也不是替代 [长周期主执行计划](/Users/abab/Desktop/Agent-Orchestratoar/docs/process/长周期主执行计划.md)。当前主线仍然是：

- 先保持本仓库内部默认可用
- 再保持完整 v1 的端到端约束
- 后续只在 v1.x 中继续扩展 provider、runtime、benchmark 广度

## 当前判断

当前仓库已经达到完整 roadmap v1 的产品基线：

- `team start / status / revise / approve / execute / inspect` 可作为默认内部工作流连续使用
- approved plan 正式驱动 execution，并在 run artifact 中保留 session、topology、provider/runtime provenance
- direct run 与 team run 共享扩展后的 `ExecutionContract` 核心结构
- author / reviewer / runtime fallback artifact 已记录 preferred、actual、fallback source、reason、detail
- recovery guidance 已能区分 retry、resume、inspect-before-rerun、escalate
- topology 与 review policy 已输出结构化 signals 和 execution config
- changed-file doc sync、file-header dependency、module manifest、root map、operator runbook signal 已纳入 compliance
- managed hook marker 与 pre-commit drift 已有结构化 warning、required action 和 repair command
- evidence harness 已有稳定 schema、固定 benchmark cases、scenario aggregates 和 reportable summary

完整 v1 的定义不是 provider/runtime 功能大而全，而是：

- 可运行
- 可恢复
- 可解释
- 可被 hook/compliance 约束
- 可用 evidence 证明 team workflow 的收益

## V1 验收清单

### 1. 默认工作流可信度

状态：

- v1 已完成

验收：

- `team` happy path 不需要手改 JSON
- failed / pending delegated review 有标准恢复入口
- compliance、execution blocker、human escalation 都会进入 operator guidance
- `status_summary` 暴露 `recovery_semantics`

### 2. provider / runtime fallback policy

状态：

- v1 已完成

验收：

- author、reviewer、runtime 三类选择均有 preferred / actual / fallback artifact
- runtime adapter 能记录 unsupported provider hint、adapter missing、provider unavailable
- direct run metadata、approved-plan execution contract、job lifecycle 均可携带 fallback 信息

### 3. topology 与 review policy

状态：

- v1 已完成

验收：

- topology signals 包含风险、并行度、依赖形态、复杂度
- review policy 包含 `standard`、`adversarial_required`、`human_escalation_required`
- review policy 带 execution config，不只是文本说明

### 4. doc-code sync 与 hook enforcement

状态：

- v1 已完成

验收：

- changed-file compliance 能识别 header、dependency、module manifest、root map、runbook signal 漂移
- managed hook marker 缺失、损坏、pre-commit 未安装或漂移会给出 warning
- compliance contract 提供 required actions 与 repair commands
- hook enforcement 对当前仓库形成默认约束，同时不阻断无关文件提交

### 5. direct/team artifact convergence

状态：

- v1 已完成

验收：

- direct run 与 team run 共享 `ExecutionContract` 字段：
  - topology
  - provider recommendation
  - review policy
  - fallback policy
  - compliance snapshot
  - gating
- presenter / inspect / summary 能围绕同一结构解释 provenance 与恢复动作

### 6. evidence 与收益证明

状态：

- v1 已完成

验收：

- evidence payload 有 `schema_version`
- benchmark cases 覆盖 standard、followup、high risk、parallel
- report 输出 scenario aggregates、signal counts、benefit score
- evidence 同时记录 provenance、recovery、doc-sync、fallback signals

## 最终 v1 gate

完整 v1 收尾必须通过：

```bash
pytest
env PYTHONPATH=src python -m agent_orchestrator.cli team check-compliance
```

并检查：

```bash
git status --short
```

允许存在的非 v1 阻塞项：

- 本地 `.DS_Store` 未跟踪文件
- 外部真实 Claude/Codex 登录状态
- provider shell、bridge product、session manager 等 v1 明确不覆盖的能力

## v1.x 后续 backlog

后续不是 v1 阻塞项，按价值继续推进：

1. 接入真实 provider health check 到更多 CLI 入口
2. 扩展 evidence benchmark 的真实任务样本数
3. 增加更细的 doc-code sync 覆盖，例如跨包 root map 层级
4. 把 review policy 配置暴露为受控 CLI 参数
5. 为带空格文件名改造 hook 脚本的 staged file 遍历方式
6. 将 evidence summary 自动写入阶段报告

## 当前结论

当前仓库已经不再停留在 `internal default`。

它已经达到完整 v1：默认 workflow 可运行，策略可解释，恢复可操作，文档同步可检查，hook 可约束，evidence 可报告。

# 完整计划待办清单

## 目的

这份文档用于说明：

- 当前仓库在达到 `internal default` 之后
- 距离“更完整的 roadmap v1 / 完整计划”还差什么
- 后续应该按什么优先级继续推进

这不是新的主计划，也不是替代 [长周期主执行计划](/Users/abaaba/Documents/Agent-Orchestratoar/docs/process/长周期主执行计划.md)。

它的作用是把“剩余差距”整理成一份可以连续执行的 backlog，避免后续判断只停留在口头描述。

## 当前判断

当前仓库已经基本补齐 `internal default` 所需的主链路能力：

- `team start / status / revise / approve / execute / inspect` 基本可连续使用
- provider-backed review/runtime 的失败和恢复提示已明显改善
- approved-plan provenance、decision verdict、topology/provider 关键信息已能落盘
- changed-file compliance、runbook signal、basic doc sync 已形成窄门槛阻断

当前剩余差距，主要不再是“基础链路缺失”，而是以下几类问题：

- 策略广度还不够
- 恢复语义还不够深
- 文档 / hook / doc-code sync 还不够硬
- evidence 还不足以支撑更完整的产品收益证明
- direct/team 两条路径的产品收敛仍有尾差

## 执行原则

后续继续按以下优先级推进：

1. 先补会影响默认工作流可信度的硬缺口
2. 再补会影响策略解释力与恢复能力的中层缺口
3. 最后补更偏产品化、证据化、对外叙事的收尾能力

## P0：默认工作流可信度缺口

### 1. stale file-header dependency blocking

状态：

- 已完成

现状：

- 当前 compliance 已覆盖 changed-file header、module manifest、root map、operator runbook signal
- 但还没有对 stale file-header dependency 建立 blocking 规则

为什么重要：

- 这是当前 hook/compliance 层最明确的缺口之一
- 如果文件依赖声明长期漂移，文档同步机制会失去可信度

完成标准：

- `team check-compliance` 能识别 changed file 中的 stale dependency declaration
- session / project compliance 都能返回结构化 blocking reason
- `required_actions` 与 `recommended_commands` 能提供明确修复指引
- 补定点测试覆盖通过与失败分支

### 2. multi-provider fallback policy 扩展

状态：

- 第一阶段已完成

现状：

- reviewer fallback 一致性已经明显改善
- 但更广义的 multi-provider fallback policy 仍未完成

为什么重要：

- 目前的 fallback 还偏 reviewer 局部修补
- 完整计划要求 provider/runtime 推荐与真实执行路径保持更稳定的一致性

完成标准：

- author / reviewer / runtime 三类角色的 fallback 规则都可解释
- fallback source、reason、detail、preferred choice 在 artifact 中完整记录
- `team status` / `summary` / `inspect-blockers` 能稳定反映 fallback 结果
- 补回退链路与 unavailable provider 的定点测试

当前进展：

- reviewer fallback 已从“固定退到 mock”扩展为优先尝试可用 provider
- author fallback 已纳入 recommendation / decision verdict / rationale
- retry review 继续沿 recommendation 保持 provider 一致性
- runtime 级别的更广义 fallback policy 仍可继续做深

### 3. interruption-aware round recovery

状态：

- 第一阶段已完成

现状：

- 当前 session guidance、summary、next、runbook 已能提供恢复建议
- 但更细粒度的 round interruption recovery 仍未完整

为什么重要：

- 这是 planning governance 从“基本可恢复”走向“真实长期使用”的关键一步
- 如果 round 被中断后只能靠人工判断，默认工作流稳定性会下降

完成标准：

- review / adversarial review / revise 等 round 被打断后有标准恢复语义
- 可以区分 retry、resume、re-run、escalate 的触发条件
- session 状态和 CLI 提示不需要依赖人工翻 JSON 理解
- 测试覆盖多类 interruption 场景

当前进展：

- failed delegated review / adversarial review 已有标准 retry 入口
- delegated review job 处于 `pending` / `running` 时，`resume` / `status` 已能明确提示 `inspect_delegated_job`
- compliance blocking、execution blocking、required gap blocking、human escalation 已有稳定恢复分流
- 更深层的 round-level retry/escalate policy 仍可继续细化

## P1：策略层与执行拓扑缺口

### 4. richer topology policy

状态：

- 第一阶段已完成

现状：

- 当前 topology / provider policy 已可解释
- 但仍带有明显 heuristic 色彩

为什么重要：

- 完整计划不只是让 team 路径能跑，而是要让 strategy decisions 更稳定、更可复用、更可验证

完成标准：

- topology 决策因素更明确，例如任务风险、并行度、依赖形态、审查强度
- decision verdict 中的 topology rationale 更结构化
- 不同 mode 与不同任务特征下的 topology 选择有清晰测试覆盖
- operator 输出能解释“为何不是别的 topology”

当前进展：

- topology recommendation 已补充结构化 signals
- 当前选择已显式考虑风险、并行性、依赖形态、复杂度
- 并行低风险任务与高风险任务已有定点覆盖
- 更深层的 topology policy breadth 与 mode-specific expansion 仍可继续做

### 5. configurable multi-role review policy

状态：

- 第一阶段已完成

现状：

- 当前仍是固定 dual-model review 模板
- 缺少更灵活的角色/轮次策略

为什么重要：

- 完整计划里的 planning governance 不应永远锁死在单一 review 模板
- 否则策略层的扩展空间会长期受限

完成标准：

- review policy 至少在配置层可扩展，而不是写死在流程里
- 不同任务类型或风险级别可选择不同 review 强度
- artifact 能记录实际采用的 review policy
- 不破坏当前 `internal default` happy path

当前进展：

- session / approved-plan artifact 已记录结构化 review policy
- 当前可区分 `standard` 与 `adversarial_required` 两类 review 强度
- review policy 已与 topology / risk signals 对齐
- 后续仍可继续把 policy 从“记录层”推进到更完整的执行层配置

## P2：文档同步与 hook enforcement 缺口

### 6. richer doc-code sync coverage

状态：

- 第一阶段已完成

现状：

- 当前文档同步已覆盖 root map、module manifest、header contract、runbook signal 的关键窄范围
- 但更丰富的 doc/code mismatch coverage 仍未完成

为什么重要：

- 当前合规门槛已经能挡住明显漂移
- 但距离“文档与代码结构硬同步”还有一段距离

完成标准：

- changed-file doc sync 检查能识别更多实际结构漂移
- 降低“代码改了但 process docs 仍看起来通过”的漏检概率
- 维持较低噪音，不把 compliance 变成过度干扰

当前进展：

- changed-file doc sync 已不仅检查模块是否缺失，还会检查 module manifest summary 是否过时
- 文档漂移检测继续保持在 changed-file 窄范围内，避免扩大噪音
- 后续仍可继续增加更细粒度的结构漂移覆盖

### 7. automatic global map refresh semantics 扩展

状态：

- 第一阶段已完成

现状：

- 关键 team workflow transition 已会触发部分 refresh
- 但 broader task-completion refresh semantics 仍不完整

为什么重要：

- 完整计划里的文档层目标，不只是“发现漂移”，还包括“完成任务后自动刷新全局上下文”

完成标准：

- 对哪些事件触发 refresh 有稳定规则
- refresh 结果能进入 evidence / compliance / operator guidance
- 不会因为自动刷新过宽而产生过多噪音更新

当前进展：

- `approve` / `execute` 之外，`revise` 与 review retry 相关路径也已具备受控 refresh 能力
- refresh 触发规则已开始从“完成态”扩展到“关键 plan state transition”
- 仍通过项目内 `docs/process` 基线存在与否控制噪音

### 8. stronger hook enforcement

状态：

- 第一阶段已完成

现状：

- 仓库内已有受管 `pre-commit` hook source
- 但安装仍是 opt-in

为什么重要：

- 完整计划希望 hook-based compliance checks 成为真实工作流约束，而不是可选提醒

完成标准：

- hook 安装与启用路径更稳定、更不易被绕过
- 至少对当前仓库内部工作流形成更强默认约束
- 文档明确说明强制边界与例外情况

当前进展：

- `install-hooks` 现已写入 managed hook marker
- changed-file compliance 会在仓库未启用受管 hooks 时给出 warning
- hook enforcement 仍保持 non-blocking 提示，不直接破坏当前 happy path

## P3：产品收敛与互操作缺口

### 9. direct/team artifact convergence cleanup

状态：

- 第一阶段已完成

现状：

- direct run 与 team run 已共享 execution contract 核心结构
- 但 broader interoperability cleanup 仍有尾差

为什么重要：

- 完整计划要求两条路径在产品语义上逐步收敛，而不是长期形成两套近似但不一致的 artifact

完成标准：

- direct/team 在 provenance、decision contract、execution metadata 上进一步统一
- 主要 presenter / inspect / summary 逻辑不需要为两套结构分别兜底
- 文档中对两条路径差异的描述更清晰、更收敛

当前进展：

- direct run 现已补充 `approved_plan_summary` 风格摘要
- direct provenance 现已补充 `selected_topology` 与 `selected_provider_runtime`
- direct/team 两条路径在 metadata 结构上进一步收敛

### 10. roadmap / README / process 持续对齐

现状：

- 当前三份主文档已基本对齐到 dual-layer 叙事
- 但过程文档明确要求持续防止再次漂移

为什么重要：

- 完整计划的一部分就是“产品叙事不再反复切换”
- 文档漂移会直接削弱 compliance 层的意义

完成标准：

- 新增策略或流程时，同步更新三类主文档
- 不再出现 README 说已完成、process 说仍缺骨架、runbook 又是第三套说法
- 增加必要的文档一致性测试或检查点

## P4：evidence 与收益证明缺口

### 11. real-task evidence benchmark 扩展

现状：

- 当前已有轻量 evidence harness
- 已能输出结构化 case 与聚合指标

为什么重要：

- 完整计划不仅要求系统能跑，还要求能证明相比 direct run 或固定流程有收益

完成标准：

- 引入少量真实仓库任务作为固定样本
- 输出 team workflow 与 direct run 的结构化对照结果
- evidence 不只记录“是否成功”，还记录 provenance、恢复、文档同步、决策解释收益

### 12. reportable evidence format 产品化

现状：

- evidence harness 已可用
- 但还更偏内部验证工具

为什么重要：

- 如果没有更稳定的 report 形态，就很难把 evidence 用于阶段总结、回归比较和 roadmap 证明

完成标准：

- evidence 输出格式更稳定
- 支持阶段比较、基线比较或聚合摘要
- 能直接支撑 README / process 中的收益叙事

## 推荐执行顺序

建议后续按以下顺序继续：

1. `stale file-header dependency blocking`
2. `multi-provider fallback policy`
3. `interruption-aware round recovery`
4. `richer topology policy`
5. `richer doc-code sync coverage`
6. `automatic global map refresh semantics`
7. `stronger hook enforcement`
8. `direct/team artifact convergence cleanup`
9. `real-task evidence benchmark`
10. `reportable evidence format`

## 当前结论

当前仓库已经不再主要缺“能不能走通 internal default”。

后续真正的工作重心是：

- 把默认工作流继续做硬
- 把策略解释与恢复能力做宽
- 把 compliance / hook / doc sync 做深
- 把 evidence 从“有样例”推进到“能证明收益”

后续实现应继续优先避免：

- 为了扩功能而破坏当前 happy path
- 为了追求广度而重新回到大量人工判断
- 为了产品叙事而弱化当前已经形成的结构化 artifact 与合规约束

# Goal Mode Template Detail

## Goal Intent

本 goal 的目的，是把 `[当前能力 / 当前状态]` 推进到 `[目标能力 / 可验收状态]`，而不是只增加局部 helper、文档说明或表面产品化。

这里的“完成”不是泛泛地指 `[某个模块存在]`，而是指它能在同一个受治理语义下完成下面这条链路：

- 接收明确输入；
- 选择或生成必要上下文；
- 执行核心动作；
- 产生可追踪状态或制品；
- 处理失败、暂停、恢复或边界情况；
- 把结果投影到 operator 可见面；
- 通过测试、命令或等价证据证明主链路有效。

本 goal 必须防止陷入“抽象越来越多、文档越来越漂亮、但主链路没有变强”的局部最优。

## Target Outcome

完成后，当前项目应从：

- `[当前弱点 1]`
- `[当前弱点 2]`
- `[当前弱点 3]`

推进到：

- `[目标能力 1]`
- `[目标能力 2]`
- `[目标能力 3]`

剩余差距应主要体现在：

1. `[明确允许留到后续 goal 的产品厚度 / 生态 / 性能 / 泛化能力]`
2. `[明确允许留到后续 goal 的非核心能力]`

而不应主要卡在：

1. 主链路不稳定；
2. 核心能力只存在于 helper 或 demo；
3. 失败、暂停、恢复或边界语义不清晰；
4. 结果事实无法稳定投影到用户可见面；
5. 相关文档、代码和测试无法互相对照。

## Scope Boundary

本 goal 只包含以下四个实现部分，且都必须完成。

### P0 Foundation / Baseline

建立最小但真实的基础链路。

必须完成：

- 输入能进入正式主链路；
- 主链路能跨过一次以上步骤，而不是只生成计划；
- 执行过程能产出结构化状态、制品或等价事实；
- 状态或结果能被后续步骤读取；
- 至少一个真实任务类可以推进到完成或明确停止。

### P1 Main Capability

把核心能力接入主链路，而不是停留在 helper、脚本或文档概念中。

必须完成：

- `[核心动作 1]` 真实进入主路径；
- `[核心动作 2]` 真实执行，而不是只记录预期；
- 执行结果有可追踪的 artifact、diff、日志、状态或等价记录；
- 失败或不满足条件时能进入修复、重试、暂停或明确停止；
- 存在测试、命令或等价直接验证。

### P2 Recovery / Governance / Edge Case

让失败、暂停、恢复、审批或边界情况成为正式语义。

必须完成：

- 风险或阻塞状态可以被明确识别；
- resume / retry / repair / approval / stop 中至少一种恢复语义可用；
- 恢复或停止基于明确状态，而不是重跑一切或依赖人工读源码；
- 恢复前后的状态差异可见；
- 不绕过既有 control plane、审计、审批或证据边界。

### P3 Operator Visibility / Evidence

让执行事实能被 operator 直接看见和判断。

必须完成：

- CLI summary 能显示当前进度、结果、阻塞原因或下一步；
- UI、workspace index、runtime payload、artifact 或 docs 至少一个面能显示同一事实；
- 至少两个 surface 共享同一条事实链或可对照证据；
- 用户能区分“正在执行”“已暂停”“可恢复”“已完成”“已停止”中的相关状态；
- 事实投影与运行状态一致，不互相打架。

## Global Stopping Criteria

只有以下条件全部满足，goal 才能停止：

1. `P0`、`P1`、`P2`、`P3` 四部分都已完成。
2. 至少一类真实任务可以在当前主链路里稳定跑通或明确停止。
3. 核心能力不再只是局部 helper、脚本、文档或一次性 demo，而是主链路语义。
4. 失败、暂停、恢复、审批或边界情况至少有一种被正式处理。
5. CLI / UI / workspace index / runtime payload / artifact / docs 至少两个面能看到同一事实链。
6. 相关文档、代码和测试可以互相对照，说明这个能力不是一次性演示。
7. 不得继续把目标扩展为：
   - 全面追平某个参考项目的所有 UI / 插件 / 生态能力；
   - 大规模重构整个仓库结构；
   - 为了更漂亮的抽象继续拆分执行层；
   - 做与本 goal 主链路无关的产品化扩张。

## Phase Acceptance Criteria

### P0 Acceptance

`P0` 只有在以下条件全部满足时才算完成：

1. 输入能进入正式主链路。
2. 主链路能跨过一次以上步骤。
3. 执行过程能产出结构化状态或等价事实。
4. 状态或结果能被后续步骤读取。
5. 存在测试、命令或等价直接验证。

### P1 Acceptance

`P1` 只有在以下条件全部满足时才算完成：

1. 核心动作真实进入主链路。
2. 结果真实产生，而不是只记录预期。
3. 失败能进入修复、重试、暂停或明确停止。
4. 结果能以 artifact、diff、日志、状态或等价形式追踪。
5. 存在测试、命令或等价直接验证。

### P2 Acceptance

`P2` 只有在以下条件全部满足时才算完成：

1. 风险、阻塞或边界状态可以被识别。
2. 恢复、重试、审批或停止语义至少一种可用。
3. 恢复或停止有明确状态支撑。
4. 恢复前后事实能在记录中区分。
5. 存在测试、命令或等价直接验证。

### P3 Acceptance

`P3` 只有在以下条件全部满足时才算完成：

1. 至少两个 surface 能看到同一执行链或同一证据链。
2. 阻塞、完成、暂停、恢复等相关状态可区分。
3. operator 无需翻源码就能判断当前是否还在可控范围内。
4. 事实投影与运行状态一致，不互相打架。
5. 存在测试、命令或等价直接验证。

## Anti-Local-Optimum Guardrail

以下情况一旦成立，就应停止当前 goal，而不是继续局部优化：

1. 核心闭环已经成立，但后续工作主要变成增加更多工具名、字段名或文档段落。
2. 代码已经能完成目标任务，但继续优化只是让抽象更漂亮。
3. 继续工作主要是在做 UI 漂亮化、命名统一化或目录重排。
4. 当前能力已经足够支撑真实任务，剩余差距主要是产品厚度、生态规模或泛化广度。
5. 新增工作无法直接映射到 `P0`、`P1`、`P2`、`P3` 中至少一项验收标准。

如果剩余工作大多属于下面这些，应进入下一个 goal，而不是继续本 goal：

- 更强的通用 agent 生态；
- 更大的插件系统；
- 更厚的产品界面；
- 更广泛的任务类型覆盖；
- 更深的性能优化；
- 更漂亮但非必要的架构抽象。

## Required Verification Evidence

goal 完成前，必须至少具备以下一种或多种直接证据组合：

- 自动化测试；
- 集成测试；
- CLI 级运行验证；
- runtime / workspace / recovery 状态直接检查；
- artifact / evidence 输出证明；
- UI / summary 投影检查；
- 文档与实现/测试之间的交叉引用。

不能只用以下弱证据宣布完成：

- “看起来已经能跑”；
- 只有 helper 存在但主链路没接入；
- 只写文档没落代码；
- 只提升抽象层次但没有真实任务闭环；
- 只修局部失败而没有形成可停止的终点；
- 单个 mock 测试通过但没有证明主路径；
- 把未验证的后续工作写成已经完成。
